
## 统一数据管理
- 数据初始化
  - 一次性元数据录入：wiki， google doc, 使用 AI agent 解析知识 
    - glossary domain desc definition
  - 常规API 数据同步：table metric lineage、 pipeline（code logic）
  - 利用AI： 智能对齐口径与 冲突检测，比如 GMV 的定义
    - **组织**上解决业务指标的定义（包括底层表 -> 公式）统一
- 数据治理
  - 统一“语言”：
  - PII 脱敏
  - 监控指标：定义质量规则（完整性、准确性、一致性、及时性，数据去重）
  - 业务概念的定义，比如每个团队都在算GMV 但口径不同，这时候就不能都用GMV
  - 面向层次开发，减少烟囱式开发（每个层次都要有架构师负责统一规划，审批，还是组织架构的问题）
    - 这里可以借助 AI 初审，比如 命名， 字段描述，校验，表重复
    - 比如 ADS 不能直接去 创建 DWS 或更底层的数据
    ```text
       层级   名称     核心职责                     建模方式
       ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       ODS    贴源层   原始数据接入，保留历史快照   保持源系统结构
       DWD    明细层   数据清洗、标准化、轻度汇总   维度建模（星型/雪花）
       DWS    服务层   面向主题的宽表聚合           宽表设计、预聚合
       ADS    应用层   直接支撑业务应用             按需定制
       
       {分层前缀}_{业务域/主题}_{数据内容/描述}_{粒度/周期}_{更新策略/后缀}
        名称： dws_usr_user_active_1d
        解读： DWS层 + 用户域 + 用户活跃主题 + 最近1天统计。
    ```

  - 性能监控：
    - SQL 监控，是否扫描全分区
    - 数据倾斜等特殊处理
- 指标
  - Semantic Model --> 应对不同的BI tools，以及 MCP
  - {业务限定}_{统计粒度}_{原子指标}_{时间周期}
    - 原子指标 = 业务过程 + 度量 + 聚合逻辑
    - 派生指标 = 原子指标 + 修饰词 + 时间周期
    - 复合指标 = 派生指标之间的运算

| 业务场景 | 命名示例 | 解读 |
| :--- | :--- | :--- |
| 交易 | `pay_ord_cnt_1d` | 最近1天的支付订单数量 |
| 用户 | `new_usr_cnt_td` | 历史累计的新增用户数量 |
| 流量 | `ios_page_view_cnt` | iOS端的页面浏览次数 (默认通常指当天或实时) |
| 财务 | `gmv_amt_daily` | 每日的商品交易总额 |
| 物流 | `deliver_dur_avg_7d` | 最近7天的平均配送时长 |
