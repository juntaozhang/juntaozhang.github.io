# uber
**功能实现**：
- 乘客下单 → 快速找到周围可用司机 → 派单 / 调度

## 系统的规模
- 司机
  - 车辆持续上报 GPS：5秒一次
  - 在线车辆数量：10万，每天能接20单
  - 上报内容：司机id/经纬度/状态/时间戳 100~200B
  - 所以司机实时位置 20M（热数据）
  - 上报 QPS：2万条/s
    - 每天：20亿
- 用户
  - 每天有200万单成交，单据平均在12小时，50 TPS
  - 峰值是平均值的 50～100 倍
    - 峰值：5000 TPS

核心设计目标：


| 指标       | 要求                    |
| ---------- |-----------------------|
| 检索延迟   | P99 < 50ms，P50 < 10ms |
| 并发支撑   | 检索QPS 10万+            |
| 数据一致性 | 位置更新延迟 < 1s           |
| 可用性     | 99.99%（全年故障< 1h）      |
| 扩展能力   | 支持在线车辆从10万扩到50万       |

下单延迟，体感30s：
- 0～0.1s：系统查到附近司机（架构设计的能力）
- 0.1～10s：给第一批司机发单，等待回应
- 10～20s：没人接，扩大范围发第二批
- 20～30s：终于有司机接单，开始导航接驾


## 实现附近司机的快速检索
从10万司机中，找出距离user方圆5公里的司机：
[geohash](../../../../../../../misc/geohash.md)
期初是用geohash 搜索，然后找到司机之后粗糙计算直线距离的最近的top 10，
然后派发给司机。\
这里的要不要计算ETA，取决于业务场景。（反正也是调用地图公司的ETA接口）

司机接单之后，系统会根据司机的位置，把 ETA（预计接驾时间）返回给乘客。

解决 Geohash 的边界、精度与邻接问题而诞生的现代地理网格系统：
- S2 是 Google 开发的球面四叉树空间索引
- H3 是 Uber 开发的六边形层次空间索引
