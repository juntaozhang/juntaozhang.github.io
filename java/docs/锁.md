# Java 锁机制

## 阻塞 vs 非阻塞

**关注线程是否挂起等待**

| 类型 | 说明 |
|------|------|
| **阻塞（Blocking）** | 操作调用时，线程会停在那里等待结果返回，不能做其他事 |
| **非阻塞（Non-blocking）** | 操作调用时，线程不会等待，立即返回，结果可能未就绪 |

阻塞：当前线程在等待某个资源或事件（如 I/O、锁、信号量）完成时，不能继续执行后面的操作。

阻塞会涉及到线程上下文的切换，让出 CPU。

## 同步 vs 异步

**关注调用者控制权**

| 类型 | 说明 |
|------|------|
| **同步（Sync）** | 调用方必须等待操作完成，才能继续执行后续逻辑 |
| **异步（Async）** | 调用方发出操作后无需等待结果，可以继续执行，结果通过回调/事件/消息等通知回来 |

### 同步阻塞

最常见的模式，调用方等待操作完成，线程被阻塞。

### 同步非阻塞

Java NIO 是通过轮询检查实现同步非阻塞。底层使用 `epoll` 等机制。

### 异步非阻塞

事件驱动 + 回调通知：注册 IO 事件到操作系统的事件通知机制中，让内核在数据就绪时通知用户线程。

底层使用 `io_setup`、`io_submit` 等系统调用。

### `reader.readLine()` 是阻塞，`channel.read(buffer)` 是非阻塞，原因？

- `reader.readLine()` 属于 Java BIO（阻塞 IO）API，它直接基于阻塞式 `InputStream` 实现，会调用操作系统的阻塞 IO 接口，如 `read()`
- `channel.read(buffer)` 属于 Java NIO（非阻塞 IO）API，底层通道已配置为非阻塞模式，会调用操作系统的非阻塞 IO 接口，如 `recv()`，不会阻塞线程

## AQS（AbstractQueuedSynchronizer）原理

AQS 内部维护一个 `state` 状态位，尝试加锁的时候通过 CAS（CompareAndSwap）修改值：

- 如果成功设置为 `1`，并且把当前线程 ID 赋值，则代表加锁成功
- 一旦获取到锁，其他的线程将会被阻塞进入阻塞队列自旋
- 获得锁的线程释放锁的时候将会唤醒阻塞队列中的线程
- 释放锁的时候则会把 `state` 重新置为 `0`，同时当前线程 ID 置为空

## `ReentrantLock` vs `synchronized`

### `synchronized`

`synchronized` 的锁升级过程：偏向锁 → 轻量级锁 → 重量级锁的多级优化。

| 锁级别 | 触发条件 |
|--------|----------|
| **偏向锁** | 默认开启，首次由某线程访问对象时加上 |
| **撤销偏向 → 升级为轻量级锁** | 其他线程访问了已偏向的对象，触发偏向撤销 |
| **轻量级锁 → 升级为重量级锁** | 多线程并发争抢，CAS 自旋多次失败 |

### `ReentrantLock`

`ReentrantLock` 基于 AQS + CAS + `park`/`unpark` 实现。

### CAS + `park`/`unpark` 与 monitor 的重量级锁主要开销差异

CAS + `park`/`unpark` 是用户态的锁控制机制，使用原子操作与 JVM 控制的挂起/唤醒，在性能和控制能力上远优于传统重量级锁；而 monitor 的重量级锁是 OS 层面调度，依赖内核调度器的抢占式调度。

#### "依赖内核调度器"

表示线程的调度、挂起、恢复等操作是由操作系统内核（比如 Linux 的 scheduler）来决定的，用户态代码没有控制权。

操作系统会"强制打断"当前正在运行的线程，即使它没阻塞/没主动释放 CPU，然后调度另一个线程运行。这是一种非合作式（non-cooperative）的调度方式。

## 锁优化

### 锁粗化

优化前：

```java
for (int i = 0; i < 1000; i++) {
    synchronized (lock) {
        buffer.append("a");
    }
}
```

优化后：

```java
synchronized (lock) {
    for (int i = 0; i < 1000; i++) {
        buffer.append("a");
    }
}
```

当编译器发现多个连续的小范围加锁操作针对的是同一个对象锁，且没有其他线程介入，会将这些小锁合并为一个更大范围的锁，减少加解锁频率。

### 锁消除

当编译器通过逃逸分析发现某个加锁对象不会被多个线程共享，那么该锁是无意义的，就可以完全消除锁操作。

#### 逃逸分析

只有无逃逸对象才能被锁消除。

| 逃逸类型 | 说明 |
|----------|------|
| **无逃逸** | 对象只在当前方法使用 |
| **方法逃逸** | 对象作为参数传出 |
| **全局逃逸** | 对象被赋值到全局变量中 |

## 公平锁 vs 非公平锁

### 公平锁

公平锁是指多个线程按照申请锁的顺序来获取锁。

- 线程必须判断队列中是否有人排在前面
- 只有队头线程才有资格尝试获取锁
- 保证先请求先执行（FIFO）

**优势：**

- 保证线程"公平"
- 不会饿死
- 更适合实时系统、调度器、资源竞争敏感系统

**劣势：**

- 线程被严格限制排队 → 每次都要上下文切换
- 吞吐量比非公平锁低
- 系统调度、缓存局部性更差

### 非公平锁

非公平锁是指多个线程获取锁的顺序并不是按照申请锁的顺序，有可能后申请的线程比先申请的线程优先获取锁。

在**非公平锁**中（如默认的 `ReentrantLock(false)`）：

- 每个线程进来都尝试直接获取锁（CAS）
- 即使有线程在队列中排着队，也可以插队成功
- 如果你刚好抢到了，就省去了排队的上下文切换

**优势：**

- 吞吐量高
- 锁释放 → 更快找到"可用线程"
- 减少线程切换时间（context switch）
- 适合"短时间持有锁 + 高竞争"场景

**劣势：**

- 某些线程可能永远抢不到锁（饿死）
- 不公平、不可预测
- 会影响响应时间的一致性

### 为什么非公平锁吞吐量会提高？

非公平锁的优点在于吞吐量比公平锁大。对于 `synchronized` 而言，也是一种非公平锁。

公平锁相比非公平锁，核心就多了一个 `hasQueuedPredecessors()` 判断——用来"看看自己前面有没有排队的人"。

- 判断当前线程是不是等待队列的"第一个人"
- 如果你前面还有别的线程，就返回 `true` → 表示你不能抢锁，得排队
- 如果你就是队首，就可以抢锁

## 共享锁 vs 互斥锁

读写锁在 Java 中的具体实现就是 `ReentrantReadWriteLock`。

| 类型 | 说明 |
|------|------|
| **共享锁（Shared Lock）** | 允许多个线程"共享"读访问 |
| **互斥锁（独占锁、Exclusive Lock）** | 一次只能有一个线程持有，其他线程必须等待 |

## 自旋锁 vs 阻塞锁

### 阻塞锁

阻塞锁可以说是让线程进入阻塞状态进行等待，当获得相应的信号（唤醒、时间）时，才可以进入线程的准备就绪状态，准备就绪状态的所有线程，通过竞争进入运行状态。

Java 中，能够进入/退出、阻塞状态或包含阻塞锁的方法有：`synchronized` 关键字（其中的重量锁）、`ReentrantLock`、`Object.wait()`/`notify()`、`LockSupport.park()`/`unpark()`。

### 自旋锁

自旋锁就是 CAS。
