# JVM 内存结构

JVM 内存结构就是 JVM 运行 Java 程序时，在内存里划分的几个不同区域，每个区域负责不同的工作，比如存储类信息、对象、栈帧、执行线程等。

## 线程私有区域

| 区域 | 是否线程私有 | 说明 |
|------|------------|------|
| **程序计数器（PC）** | 是 | 记录当前线程执行到哪条字节码 |
| **Java 虚拟机栈（Stack）** | 是 | 每个方法调用都会创建一个栈帧 |
| **本地方法栈（Native Stack）** | 是 | 调用 C/C++ 本地方法时用 |

### 虚拟机栈（Stack）

虚拟机栈是**每个线程自己的方法调用记录本**。

```java
public int add(int a, int b) {
    int sum = a + b;
    return sum;
}
```

- 每次方法调用，就在栈里压一个"栈帧"
- 方法结束后对应的栈帧出栈
- 里面记录了：局部变量、操作数、返回地址等

> **注意**：栈空间用完就报 `StackOverflowError`

### 本地方法栈（Native Method Stack）

当你使用 JNI（Java Native Interface）去调用 native 方法时，使用这个栈。比如：`System.loadLibrary()` 之类。

## 线程共享区域

| 区域 | 是否线程私有 | 说明 |
|------|------------|------|
| **堆（Heap）** | 否 | 所有对象实例、数组的内存分配区域 |
| **方法区（Metaspace）** | 否 | 存储类的结构信息，如常量池、方法代码等 |

### 方法区（Metaspace）

方法区是**类的信息仓库**：

- 存储类结构信息（类名、字段、方法、常量池等）
- Java 8 以前叫**永久代（PermGen）**
- Java 8 开始改成使用**本地内存（Metaspace）**，不会再受 JVM 内存限制

> **注意**：类太多或加载类太频繁会导致 `OutOfMemoryError: Metaspace`

## 线程的工作内存与线程栈的关系

主内存复制来的变量是放在线程栈里的吗？

### JMM 中的"工作内存"并不等同于 JVM 中的"线程栈"

```java
static int shared = 0;

public static void main(String[] args) {
    new Thread(() -> {
        int a = shared; // 从主内存加载到"工作内存"
        // ...
    }).start();
}
```

- `shared` 是主内存中的共享变量
- 线程读取它的值，会复制到自己的工作内存（可能是寄存器或临时缓存）
- `a` 是方法内的局部变量，会放在线程栈中
- 这两者不是一回事，但都属于线程私有的数据空间

### 为什么不能说工作内存就是线程栈？

- 栈里只有局部变量和执行上下文
- 共享变量的副本（特别是 `volatile` 等受可见性控制的变量）不一定放在栈里
- 而是 JVM/JIT/CPU 可能选择更高效的地方，比如：
  - CPU 缓存（比如 L1 Cache）
  - 虚拟寄存器
  - 优化后的内联结构

#### 虚拟寄存器

虚拟寄存器在 JVM 字节码/JIT 编译器层面是一种抽象概念，它有可能映射到真实 CPU 寄存器中，但也可能映射到内存（栈帧中的 slot 或栈空间）。

#### 优化后的内联结构

```java
class Data {
    int x = 0;
}

Data d = new Data();
d.x = 42;
```

经过逃逸分析优化后：

```java
int x = 42; // 不再有 Data 对象存在，也没有堆分配
```

此时，`d.x` 被直接"展开"成一个局部变量 `x`，这个变量确实在线程栈中。所以它**既不在主内存，也不在工作内存的共享副本结构中**！

## 寄存器 vs CPU Cache

- **寄存器**是 CPU 内部最小最快的"数据寄存点"，直接参与计算
- **L1/L2 Cache** 是在内存和 CPU 之间做加速的"中间缓冲层"
