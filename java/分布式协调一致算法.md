# 分布式协调一致算法

## Paxos

### 角色

| 角色 | 说明 |
|------|------|
| **Proposer** | 理论上任何节点都可以是 Proposer |
| **Acceptor（陪审团）** | 受提案并表态的节点 |
| **Learner** | 用于"学习最终被选定的值"，但实际系统中，Acceptor = Learner 是最常见的做法 |

一次 Paxos 实例，只为一个提案位置（一个 key、一个 slot）达成一次共识，且值一旦决定就不能更改。如果涉及到多次提案共识，涉及到 Multi-Paxos。

### 提案编号

Paxos 中每个 Proposer 会构造由自己编号 + 本地递增号组成的 proposal ID，来保证它是全局唯一且递增的。

### 两阶段流程

| 阶段 | 说明 |
|------|------|
| **第一阶段：Prepare** | 询问"我能不能提编号 n 的提案"并获得承诺 |
| **第二阶段：Accept** | 发送具体值 (n, v) 请求 Acceptor 接受 |

### Read

向多个 Acceptor 或 Learner 发起读取请求，收集多数响应，再选出一致的值。

### 值继承原则

如果 Acceptor 已经接受过某个提案，那么在新的 Prepare 阶段，它会把这个已接受的值返回给 Proposer，Proposer 在 Accept 阶段必须继承这个值（而不是自己提议的新值），以保证一致性。

## Multi-Paxos

每个 slot 独立运行一次 Paxos，但多个 slot 组合起来，就能驱动一个有序的状态机复制（SMR）。

Single Paxos 每次都要跑完整的两阶段（Prepare → Accept）。Multi-Paxos 在选出一个稳定的 Leader 后：

- 只有 Leader 发起提案，这样就可以保存顺序了
- 减少 Prepare 阶段：新的 slot 可以直接进入 Accept 阶段（因为 Leader 已经和多数 Acceptor 建立了 promised 关系）

这样连续提交多个值时，大部分 slot 只需要一轮网络交互（Accept 阶段），大幅降低延迟。

## Raft

### 核心概念

| 概念 | 说明 |
|------|------|
| **任期（Term）** | 每个任期最多一个领导者，任期号全局递增，用于标识日志的"新鲜度" |
| **角色** | 集群中每个节点只能是 Leader、Follower 或 Candidate 中的一种，同一时刻最多存在 1 个 Leader |

Raft 协议通过"任期"划分时间线（类似逻辑时钟），每个任期从一次选举开始，可能包含"选举阶段"和"领导阶段"，任期号单调递增。

### Leader 选举

```
Follower 超时 → Candidate 请求投票 → 获多数票 → Leader
```

### 日志复制

```
客户端写请求 → Leader 写本地日志
Leader → AppendEntries → Follower 写日志
Follower 返回 ACK
Leader 获多数 ACK → 标记日志已提交
```

### 日志提交

```
Leader 应用到状态机 → 返回客户端
Follower 应用到状态机 → 状态一致
```

## CAP 理论

| 特性 | 英文 | 说明 |
|------|------|------|
| **C** | Consistency（一致性） | 所有节点在同一时间看到的数据必须完全一致。例如读操作总能读到最新写入的数据（线性一致性） |
| **A** | Availability（可用性） | 系统对每个请求都能在有限时间内返回响应（不一定是最新数据，但不能拒绝或超时） |
| **P** | Partition tolerance（分区容错性） | 当网络分区（节点间通信中断）发生时，系统仍然能继续运行 |

### Raft 满足 CP + partial A

满足 P（分区容错性），同时选择 C（强一致性），在发生网络分区时牺牲一部分 A（可用性）。

```
正常：  A <--> B <--> C <--> D <--> E

分区后：
分区1: A, B, C   (3 个节点) → 有多数，能选 Leader，继续服务
分区2: D, E      (2 个节点) → 没多数，暂停写入，防止产生冲突
```

### MongoDB / Cassandra 满足 AP

满足 AP，不满足 C。

```
正常：  A <--> B <--> C <--> D <--> E

分区后 1 和 2 不能通信：
分区1: A, B, C   (3 个节点) PUT user123.name = "Tom"
分区2: D, E      (2 个节点) PUT user123.name = "Jerry"
```

同一个 key 在两个分区都有不同版本，系统必须进行冲突解决，通常使用 last write wins。

## eBay BASE 理论

| 特性 | 英文 | 说明 |
|------|------|------|
| **BA** | Basically Available | 基本可用 |
| **S** | Soft state | 软状态 |
| **E** | Eventual consistency | 最终一致性 |

ACID 追求强一致性和事务的严格可靠性，而 BASE 则以可用性和灵活性为核心，牺牲部分强一致性以换取更高的性能和容错性。

**电商下单场景示例：**

- **可用性优先**：用户操作无阻塞，系统始终能接收并响应下单请求，不会因库存检查延迟而卡顿
- **短暂不一致**：可能存在"用户看到下单成功，但实际库存不足"的软状态（通常持续几秒到几分钟）
- **最终一致**：通过异步补偿、状态校验、退款等机制，最终确保"无库存订单不生效，已支付款项必退回"
